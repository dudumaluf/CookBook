# Glossary

When in doubt, look here first. If a term you needed is missing, add it in the same PR.

## Domain

- **Asset** — a typed entity stored in the library. Each asset has a `kind` (Slice 2 ships `image`; future kinds: `imageGroup`, `soulId`, `moodboard`, `product`, `characterSheet`, `video`, `audio`, `3dObject`). Assets are draggable onto the canvas, where they spawn the matching node — preserving an `assetId` link so library edits propagate. See `src/types/asset.ts`.
- **AssetScope** — `"global" | "project"`. `global` assets are visible across all projects (Soul ID models you reuse, a moodboard you like for every commercial). `project` assets live with the current project only. Duplicating a project must reuse asset ids — never copy the underlying blobs. See ADR-0018.
- **ImageAssetSource** — discriminator on `ImageAsset`: `{ type: "remote"; bucket; key; url; mime; sizeBytes }` (primary path — bytes in Supabase Storage, URL is the CDN-cacheable public URL) or `{ type: "url"; url }` (paste-a-URL escape hatch). See ADR-0018b.
- **`cookbook-assets` bucket** — Supabase Storage bucket where uploaded images/videos/audio land. `public: true`, MIME-allowlisted (`image/*`, `video/*`, `audio/*`), **750 MB** server cap (global Storage limit raised to 1 GB to clear it). Provisioned via the `cookbook_assets_bucket` migration; the media MIME types + 500 MB cap were added when the media arc shipped (the bucket originally allowed images only at 30 MB). NOTE: the project's *global* Storage "Upload file size limit" (Dashboard → Storage → Settings) must be ≥ the bucket cap for large uploads to succeed — the effective limit is `min(global, bucket)`.
- **Supabase client** — `src/lib/supabase/client.ts`. Singleton browser client. Reads env vars; throws a clear error if they're missing. `auth.persistSession: false` until we have actual auth.
- **`uploadImageAsset(file)`** — `src/lib/library/upload-asset.ts`. File → unique key (`images/<8-hex>/<safe-filename>`) → Supabase upload → public URL. Returns the descriptor that becomes a `remote`-source asset. Throws on failure.
- **`deleteAssetObject(bucket, key)`** — companion cleanup. Best-effort (logs, doesn't throw) because a transient blip on cleanup shouldn't strand the UI.
- **Import pipeline** — `src/lib/library/import-files.ts`. Single chokepoint used by every upload entrypoint: the Library header's `UploadAssetButton`, the LibraryPanel surface drop zone, and the Image node's empty-state upload zone. Enforces image-only MIME + 25 MB per-file cap; returns `{ created, errors, ids }` so callers can toast once for the whole batch.
- **Library actions** — `src/components/library/library-actions.tsx`. Two split-action buttons in the LibraryPanel header. `UploadAssetButton` (`+`) fires the OS file picker directly — no popover. `AddAssetUrlButton` (link icon) opens a tiny URL-only popover for the rare paste case. Both feed the import pipeline (upload) / asset-store (URL).
- **AssetCard** — draggable thumbnail tile inside `LibraryContent`. Uses HTML5 DnD with the custom `application/x-cookbook-asset` MIME so OS files / foreign URLs don't trigger the canvas drop handler. Hover reveals a Delete affordance. Thumbnail src is `source.url` directly (sync for both source kinds).
- **assetToNode()** — `src/lib/library/asset-to-node.ts` maps each `AssetKind` to a `{ kind, initialConfig }` spawn rule. The canvas drop handler is kind-agnostic; new asset kinds register here. Image spawn rule denormalizes `source.url` into `config.url` so unlink keeps the node working standalone.
- **Link / Unlink (Image node)** — an Image node with `config.assetId` set is *linked* to a library asset: the asset's url is canonical (library re-uploads propagate), the body shows the asset name + Unlink button. Unlink swaps the asset's url into `config.url` so the node stays standalone regardless of source kind (since both `remote` and `url` carry a real fetchable URL now).
- **Image node empty state** — when neither `assetId` nor `url` is set, the body renders an interactive upload zone (click → OS file picker; drop OS image files → uploaded). The first imported file uploads through the same `import-files` pipeline → lands in the Library AND auto-links this node. A tiny "Or paste a URL ▾" disclosure sits below for the rare paste case. Free-URL previews get a corner ✕ Clear button on hover; linked nodes use Unlink instead.
- **Node deletion** — keyboard only. `Backspace` or `Delete` while a node is selected calls `workflow-store.removeNode` via a document-level handler we install in `CanvasFlowInner` (`tryHandleDeleteKey`). React Flow's built-in `deleteKeyCode` is explicitly disabled (`null`) because RF reads selection from its internal store which lags one render behind our props — doing it ourselves keeps the workflow store as the single source of truth. The handler bails out for editable targets so typing Backspace in the prompt bar or a text node doesn't wipe the canvas. No trash icon in the header (header space is occupied by the status chip). Matches Figma / Blender / ComfyUI convention.
- **Node chrome (single-surface, ADR-0021 + ADR-0027 + ADR-0028)** — every node card has the same shape: one-row header (`icon · editable title · …spacer… · status chip · ⋯ settings`, no border-bottom), a body that flows out of it visually with zero outer padding (the body itself decides what padding to use), and absolutely no footer. The `⋯` settings trigger anchors the rightmost slot whenever the node's schema declares `settings`; the status chip pops in/out beside it during a run without ever shifting it. When `schema.size.resizable` is set, a tiny drag handle sits in the matching position (bottom-right corner for `both`, right edge for `horizontal`, bottom edge for `vertical`) and persists user-resized dims into `NodeInstance.size`. Handle labels live in the dot's hover tooltip, never inline. Handle rails on each side use `flex justify-around` so N dots spread evenly across the card height (on tall nodes like LLM Text the dots roughly align with their matching body rows; on short nodes the single dot sits centred).
- **Node settings slot (ADR-0027)** — schemas opt into a settings popover by setting `settings: { Content; hasOverrides? }` on `NodeSchema`. `Content` is a React component receiving the same `NodeBodyProps` as `Body`. `hasOverrides` is a pure predicate over `config` driving the accent dot indicator. `GenericNode` (canvas-flow.tsx) reads the slot, instantiates `Content` with the live `nodeId / config / updateConfig / selected`, and forwards everything to BaseNode under the `settings` prop. Nodes without secondary knobs (Text, Image, Number) simply omit the slot — no `⋯` button renders.
- **Node size slot (ADR-0028)** — schemas opt into silhouette constraints + user-resize by setting `size: NodeSizeSchema` on `NodeSchema`: `defaultWidth`, `defaultHeight`, `minWidth`, `maxWidth`, `minHeight`, `maxHeight`, and `resizable: "none" | "horizontal" | "vertical" | "both"` (default `"none"`). `GenericNode` reads the slot + the per-instance `NodeInstance.size` override and forwards them to BaseNode as a unified `size` prop. BaseNode applies the constraints as inline CSS dim style and, when `resizable !== "none"`, renders `NodeBodyResizeHandle` in the matching position. Nodes without size constraints (no slot at all) keep the legacy 240 px min-width and no resize handle — backwards-compat preserved.
- **`NodeSettingsTrigger` (BaseNode)** — the standardised `⋯` (`lucide-react` `MoreHorizontal`) icon button that BaseNode renders in the rightmost header slot when a `settings` prop is provided. 24 px ghost button, wrapped in `Tooltip` (hover label = "Settings" or the schema-specific aria-label) + `Popover` (`@base-ui/react`, 280 px wide, `align="end"`, portalled to `document.body`). Stops React Flow drag/pan on pointer-down so a click never twitches the canvas. Renders an accent dot (`data-testid="node-settings-dot"`) in its top-right corner when `settings.hasOverrides === true`. Test selector: `data-testid="node-settings-trigger"`.
- **`NodeBodyResizeHandle` (BaseNode)** — the standardised drag handle BaseNode renders when `size.resizable !== "none"`. Wraps React Flow's `NodeResizeControl` with custom chrome: a 10×10 SVG "two diagonal lines" mark (universal corner-resize affordance) in the bottom-right for `both`; a short vertical grip line on the right edge for `horizontal`; a short horizontal grip line on the bottom for `vertical`. Subtle by default (`text-muted-foreground/40`), brighter on group-hover for discoverability. `aria-hidden` (mouse-only affordance) + `pointer-events-none` on the inner visual so React Flow's drag wrapper owns the pointer. Test selectors: `data-testid="node-resize-handle"` + `data-direction`.
- **`resizeNode(id, size)` (workflow store)** — persists user-set dimensions for a node (ADR-0028). Accepts a partial size (one or both axes), rounds to integer px (NodeResizeControl emits floats during a drag), de-dupes when the rounded value matches the existing one (avoids render churn on every `onNodesChange` tick), and strips the field entirely when both axes are undefined. Only fired by `canvas-flow`'s `onNodesChange` when React Flow reports a `dimensions` change with `setAttributes` truthy (i.e. the user actively dragged the handle, not a passive content-measurement event).
- **`LLMTextSettingsContent`** — the LLM Text-specific settings content component (renders Temperature / Max output tokens / Reasoning controls + the model-required hint). Exported from `node-llm-text.tsx`; pointed at by the LLM Text schema's `settings.Content`. Receives the same `NodeBodyProps` interface as `Body`, so it freely uses the same `updateConfig` mechanism. The popover wrapper / trigger / accent-dot machinery are NOT in this file — they live on BaseNode (ADR-0027). What lives here is *only* the controls.
- **`hasSettingsOverrides(config)` (LLM Text)** — pure predicate returning `true` iff `config.temperature !== undefined || config.maxTokens !== undefined || config.reasoning === true`. Wired into `schema.settings.hasOverrides`; the BaseNode trigger uses it to decide whether to render the accent dot. Pure → trivially testable; covered by 6 `schema.settings` cases in `node-llm-text.test.tsx`.
- **Edge selection (3.1b)** — click an edge → it's selected and renders with the accent stroke + `strokeWidth: 2.5`. Shift-click stacks edges into the selection. Backspace / Delete removes selected edges (and selected nodes) in one keystroke through the same `tryHandleDeleteKey` we wrote for nodes. `workflow-store.selectedEdgeIds` mirrors the node pattern; `removeNode` / `removeEdge` defensively scrub cascading or removed ids out of the selection set so a stale ghost never silently no-ops the keyboard.
- **Shift-drag = selection box (3.1b)** — pressing Shift toggles `nodesDraggable={false}` via a `shiftHeld` state in `CanvasFlowInner`. While Shift is held, RF treats every drag as a selection-box drag regardless of where it starts (a node can no longer "claim" the mousedown), so the L→R and R→L directions both behave. `keydown` / `keyup` sync the flag; a `window.blur` clear catches the case where the user alt-tabs mid-drag.
- **Flush body grammar** — node bodies use the card's bg (transparent / `bg-foreground/5` for focus washes), no borders on inline inputs, and a hair-thin `bg-border/30` divider (inset by horizontal padding) between sections when needed. Bodies own their own `px-3 py-…` so they can sit flush against the card edge when the design calls for it. New nodes inherit this grammar by following the examples in `node-text.tsx`, `node-llm-text.tsx`, `node-image.tsx`.
- **MODEL_OPTIONS (LLM Text)** — curated starter list of OpenRouter model ids exported from `node-llm-text.tsx`. Each entry is `{ id, label, reasoningRequired? }`. Easy to extend; if the user's config carries an id not in this list, the dropdown appends a `<option value="…">… (custom)</option>` so the value is never lost. `reasoningRequired: true` flags models that Fal rejects without `reasoning: true` (Gemini 2.5 Pro as of Slice 3.4) — drives the in-popover hint via `modelRequiresReasoning()`.
- **`modelRequiresReasoning(modelId)`** — looks up the curated MODEL_OPTIONS entry and returns `true` if the model has the `reasoningRequired` flag. Conservative — only models we've actually observed rejecting calls without `reasoning: true` are flagged. Lives in `node-llm-text.tsx`.
- **LLM Text body (ADR-0023 + ADR-0026 + ADR-0027)** — `[model chip] · [output area]` — the body row contains only the model chip; output area sits below. The **model chip** is a small inline pill (always visible — idle and post-run) that shows the curated model label + a `▾` chevron; click anywhere on the chip to open the native `<select>` over MODEL_OPTIONS. The **settings trigger** (`⋯` icon in the top-right of the node header — *not* the body) opens the settings popover (Slice 3.4 content; ADR-0027 chrome). The **output area** renders the executed text (selectable, wrapping) when `record.status === "done" | "cached"`, or a destructive-tinted `role="alert"` pill with the error message when `status === "error"`, or a one-line "Connect `user` on the left then click Run" placeholder otherwise. No inline editors anywhere — prompts come exclusively from upstream Text nodes wired to the `user` / `system` handles; references come from upstream Image nodes wired to `image`. `LLMTextNodeConfig` is `{ model: string, temperature?: number, maxTokens?: number, reasoning?: boolean }` — only `model` is required; the rest defer to provider defaults when unset.
- **LLM Text settings popover (ADR-0026, chrome relocated by ADR-0027)** — opens from the standardised `⋯` trigger in the node header (BaseNode-owned, see `NodeSettingsTrigger`). Three controls. **Temperature**: range slider `0–2 step 0.1` plus a numeric label that says "default" until touched; slider is 50 % opacity while at default so "not set" reads as visually distinct from "set to 0.7". **Max output tokens**: `MaxTokensInput` (see below) with a Reset button. **Reasoning**: plain checkbox with a hint line below — generic copy ("Enable for models that need explicit reasoning…") by default, accent-coloured warning ("This model requires reasoning to be on. Tick the box or the run will fail.") when `modelRequiresReasoning(config.model) && config.reasoning !== true`. The trigger gains an `accent`-coloured dot (`data-testid="node-settings-dot"` after ADR-0027; previously `llm-settings-dot`) in the corner when any setting is non-default — visible signal that a node has overrides without opening the popover. Settings are stripped during the v4 → v5 store migration if out of range, so a hand-edited localStorage value won't surface as a mid-run 400.
- **`MaxTokensInput`** — local-draft `<input type="number">` inside the LLM Text settings popover. Holds the typed string in component-local state so partial digits aren't snapped between keystrokes (typing "1500" through 1 / 15 / 150 stays drafted; only the final integer commits). Commits to the parent on every change that parses to a positive integer (or empty → undefined). External resets are handled by the parent passing a fresh `key` prop (`key={config.maxTokens ?? "default"}`) so the component remounts with the new `initialValue` — avoids the React 19 strict-mode-forbidden "setState in useEffect" sync pattern that a naive controlled-input approach would need.
- **Uniform port visuals (ADR-0023)** — every input / output dot renders the same shape (a 10 px colored circle with the card-background border ring). The only thing that differs is the **color**, which encodes the **datatype** (text / image / video / number / any). Whether a handle accepts multiple edges (`multiple:true` in the schema) is intentionally NOT visualized — the engine aggregates multi-edge connections transparently (`run-workflow.ts:252–274`), so the user discovers the capability by trying. Tooltip on hover shows only the label, no "· multi" suffix. Lets the canvas read like "color tells me what plugs in here; the rest just works".
- **Selection mirror** — `toFlowNode` in `canvas-flow.tsx` writes `selected: selectedIdSet.has(n.id)` so React Flow's visual selection ring matches the workflow store's `selectedNodeIds`. In controlled mode, RF only knows what we tell it via the `nodes` prop.
- **Node rename (per-instance label)** — `NodeInstance.label?: string`. Double-click the node title to edit inline. Enter or blur commits, Escape reverts, empty commit clears the label so the header falls back to the schema title. Persisted by `workflow-store.renameNode(id, label?)` which trims input and normalizes empty/whitespace/undefined to a cleared label. The title intentionally renders as a non-focusable `<span>` so a single-click bubbles through to React Flow node selection — making it focusable would silently break Backspace/Delete on selected nodes (React Flow ignores keys while focus is on a button/input).
- **Soul ID** — a Higgsfield-trained character model representing the user's likeness. Tied to a specific Higgsfield account. As of Slice 4 the library imports already-trained ones via `ImportSoulIdButton`; the M0b training flow will introduce in-app states (`SoulIDDraft` / `SoulIDReady`).
- **`SoulIdAsset`** _(Slice 4 / ADR-0029)_ — `Asset` union variant for trained Higgsfield characters. Pure reference (no bytes — Higgsfield owns the trained model). Carries `customReferenceId` (Higgsfield's UUID), `variant` (`v1 | v2 | cinema`, drives endpoint dispatch), `name`, `thumbnailUrl`. Imported via `useAssetStore.importSoulIdAsset()` (idempotent on `customReferenceId`).
- **`SoulIdRef`** _(Slice 4 / ADR-0029)_ — the `StandardizedOutput` shape for a Soul ID that flows through the graph: `{ customReferenceId, variant, name?, thumbnailUrl? }`. Emitted by the `Soul ID` node; consumed by `HiggsfieldImageGen.execute()` to drive both `custom_reference_id` (in the request body) and the endpoint dispatch (via `variant`).
- **`Soul ID` node** _(Slice 4)_ — reactive input node. Body shows the linked asset's thumbnail (or User-glyph fallback) + name (or UUID-prefix fallback) + variant chip (`SOUL 2 / SOUL 1 / CINEMA`) + Unlink button. Output handle `out` carries the `SoulIdRef`. Schema declares `size: { resizable: "horizontal" }` (single-row body).
- **`Higgsfield Soul` node (`higgsfield-image-gen`)** _(Slice 4 / ADR-0029)_ — executable image-gen node. Inputs: `prompt` (text), `soulId` (soul-id), `image` (image, optional reference). Output: `out` (image, multi). Settings popover: aspect ratio / resolution / batch size / seed / styleId / negative prompt. `execute()` dispatches via `mode` (reference > style > none) + `variant` (inherited from upstream `SoulIdRef.variant`). See **Endpoint dispatch by Soul variant** below.
- **`Image Iterator` node (`image-iterator`)** _(Slice 4 / ADR-0030)_ — reactive iterator node. `iterator: true` flag triggers engine fan-out. Bundles N upstream images (input handle `images`, multi) into a single array (output handle `out`); when wired to a single-input downstream, the engine runs the downstream once per item, in parallel up to `maxConcurrent`. Empty config in Slice 4 (placeholder for future `take(n)` / `skip(n)` / `randomize`).
- **`Export` node** _(Slice 4)_ — executable output node (no `out`). Input `in` (image, multi). For each piped-in image: fetch the URL → blob → re-upload to our Supabase bucket via `uploadImageFromUrl(url)` → create a `remote`-source `ImageAsset` named `${namePrefix} ${i+1}` (default "Generated") in the library. Sequential per-item (download is the bottleneck, not concurrency). Aborts mid-batch surface as `Saved K of N before failing: …`.
- **`ImportSoulIdButton`** _(Slice 4)_ — `library-actions.tsx`. Sparkles (✨) icon in the LibraryPanel header. Fetches trained Soul IDs via `fetchSoulIds` on each open (re-keyed via `openSeq` to avoid React's `set-state-in-effect` lint). Renders rows with thumbnail + name + variant chip + status. Idempotent on `customReferenceId`: re-clicking shows "Imported ✓" disabled. In-progress / not_ready characters surface as disabled rows with their status as a hint. Empty state copy + `missing_keys`-aware error pill.
- **Endpoint dispatch by Soul variant** _(Slice 4 / ADR-0029)_ — Higgsfield silently ignores `custom_reference_id` when the request hits the wrong endpoint for the character's trained variant. Dispatch table (`SOUL_ENDPOINT_BY_VARIANT` in `lib/higgsfield/higgsfield-api.ts`):
  ```
  v2     → /higgsfield-ai/soul/v2/standard
  cinema → /higgsfield-ai/soul/cinema   (style_id always dropped)
  v1     → /higgsfield-ai/soul/character
  none   → /higgsfield-ai/soul/v2/standard  (best-quality generic)
  ```
- **`concurrent_limit` (HTTP 429)** _(Slice 4 / ADR-0029)_ — Higgsfield enforces a 4-concurrent-per-keypair cap that surfaces as `{"detail":"Maximum number of concurrent requests (4) has been reached"}`. Detected by string-matching the `detail` and re-tagged with `code: "concurrent_limit"`. Stuck-queue jobs (timed-out without cancel reaching the server) hold slots; `scripts/cleanup-stuck.ts` is the antidote during development.
- **`HiggsfieldCallError`** _(Slice 4)_ — same shape as `LlmCallError`. `code` discriminator: `invalid_request | missing_keys | concurrent_limit | nsfw | upstream_failed | upstream_error | aborted | timeout | unknown | network`. `network` is fetch-level only.
- **Reference image caveat (Soul 2)** _(Slice 4 / ADR-0029)_ — `/soul/v2/standard` accepts `image_url` in the body but the visible influence on the output is subtle (the model leans on the prompt much more than the ref). The accepted M0d path is the **Image Describer recipe**: `[Image] → [LLM Text with vision system prompt] → text → [HiggsfieldImageGen.prompt]`. Once "save recipe as reusable node" lands, that subgraph becomes a single first-class `Image Describer` node.
- **Selection mode** _(Slice 5.5 / ADR-0031)_ — controls what an iterator node emits on a run. Six modes: `fixed` (always emit `items[cursor]`), `increment` (emit then cursor++ wrapping), `decrement` (emit then cursor-- wrapping), `random` (pick one at random; cursor reflects the pick), `range` (emit `items[start..end]` inclusive; cursor unchanged), `all` (emit every item; cursor unchanged). Implemented in `src/lib/iterators/selection-mode.ts` as a pure helper `applySelectionMode<T>({ items, mode, cursor, range?, random? })` returning `{ items, nextCursor }` — the caller persists `nextCursor` for advancing modes. Wraps modularly so cursor=N-1 + increment lands on 0 next run (matches ComfyUI's "incremental seed" mental model).
- **Iterator cursor** _(Slice 5.5 / ADR-0031)_ — 0-indexed integer in iterator config (`config.cursor`) pointing at the "current" item in `assetIds` / `texts`. Drives the body's `‹ N / M ›` chip (where N is `cursor + 1` for human readability), the thumbnail / preview shown next to it, and what `fixed` / `increment` / `decrement` emit on a run. Per-instance — duplicating an iterator gives it its own cursor. Persisted in workflow-store; advancing modes (`increment` / `decrement` / `random`) write the new cursor back via `useWorkflowStore.getState().updateNodeConfig` after the run.
- **`<IteratorCursor />` component** _(Slice 5.5)_ — `src/components/nodes/iterator-cursor.tsx`. Pure / store-less `‹ N / M ›` chip shared between Image Iterator and Text Iterator. 1-indexed counter; arrows clamp at boundaries (cursor=0 → ‹ disabled, cursor=N-1 → › disabled); `count <= 1` disables both. Stops pointer-down on the buttons (belt-and-suspenders alongside BaseNode's `nodrag` body class from ADR-0031 §5).
- **`Image Iterator` (post-5.5)** — replaces the Slice 4.4 multi-edge design with internal storage. Config: `{ assetIds: string[], cursor: number, selectionMode: SelectionMode, range?: { start, end } }`. `inputs: []` (no more `images` handle). `iterator: true` preserved so the engine fan-out branch dispatches per-item; `selectionMode: "all"` is the default and matches pre-5.5 fan-out-everything behaviour. Body shows a square thumbnail of `assetIds[cursor]` + cursor + mode chip; settings popover has the 6-mode dropdown + range start/end (when mode === `range`). `hasOverrides` lights the accent dot when mode !== `all` OR cursor !== 0.
- **`Text Iterator` (`text-iterator`)** _(Slice 5.5)_ — brand new in 5.5. Mirrors Image Iterator with `texts: string[]` instead of `assetIds`. Output `out` is `text` (multi). Body has an empty-state textarea (one-text-per-line) that splits on blur; populated state shows a `line-clamp-3` preview of the cursor item + cursor + mode chip. Settings popover carries an additional textarea synced to `config.texts` so users can edit the bag without leaving the canvas. Same `selectionMode` vocabulary as Image Iterator. Used for prompt batches that fan-out onto a single-input downstream (e.g. an LLM Text node).
- **Workflow-store v8 migration (iterator collapse)** _(Slice 5.5a)_ — for every existing `image-iterator` node: find edges with `targetHandle === "images"`, walk each edge's source, resolve upstream `image` node `assetId`s, collapse into the iterator's new `config.assetIds[]`, default `selectionMode: "all"` (matches pre-5.5 behaviour bit-for-bit), then drop the now-orphan edges from `state.edges`. Idempotent on v8 + tolerant of hand-edited fields. The text-iterator node is brand new in v8 so no migration path; the migrate still sanitises malformed text-iterator config defensively.
- **Asset multi-select** _(Slice 5.5c)_ — Finder-style selection on the library: plain click sets selection to one card, cmd / ctrl-click toggles membership, shift-click range-selects from the anchor (the last selected card) walking `assets` insertion order. State lives on `useAssetStore` as transient `selectedAssetIds` + `selectionAnchorId` (NOT persisted via `partialize` — selection is session UI). Cleared automatically after a successful drop onto the canvas (`clearAssetSelection`).
- **Multi-id drag payload** _(Slice 5.5c)_ — `{ assetIds: string[], kind }` (was `{ assetId, kind }`). Single drags carry a 1-element array so the parser sees one shape; legacy single-id payloads still parse via promotion in `parseAssetDrag` (back-compat). Dragging a card that's part of the selection ships ALL ids; dragging an unselected card resets selection to it first, then ships only its id (Finder semantics).
- **`dispatchAssetDrop`** _(Slice 5.5c)_ — `src/lib/library/dispatch-asset-drop.ts`. Pure framework-agnostic decision tree: 1 image → spawn Image node; N images → spawn pre-populated Image Iterator; drop on existing Iterator → append (de-duped); Soul IDs spawn one node per id (no iterator collapsing). Returns `AssetDropAction[]` that `canvas-flow.tsx`'s `onDrop` reduces into store mutations. Hit-test happens in the caller via `closest('.react-flow__node')[data-id]`.
- **Engine fan-out (`iterator: true`)** _(Slice 4 / ADR-0030)_ — when a node's schema declares `iterator: true` AND its output is an array AND it lands on a single-input downstream handle, `runWorkflow` dispatches the downstream once per item, in parallel bounded by `maxConcurrent` (default 4). Cache key unchanged: the aggregated output is cached by the same `computeNodeHash` recipe. Supersedes the strict-serial portion of ADR-0019; non-iterator graphs are bit-identical to before.
- **`maxConcurrent`** _(Slice 4 / ADR-0030)_ — `RunWorkflowOptions.maxConcurrent`, default 4 (= Higgsfield's keypair cap). Bound for in-flight executions during a fan-out. Per-recipe configurability lands in Slice 5+.
- **`fanOut` on `ExecutionRecord`** _(Slice 4)_ — `{ total: number, done: number }` block surfacing fan-out progress (e.g. "running 3 / 8") on the running record. Emitted on every per-item completion. Absent when the node isn't a fan-out target.
- **`uploadImageFromUrl(url, filenameHint?)`** _(Slice 4)_ — `lib/library/upload-asset.ts`. Fetches a remote URL, blobs the response, wraps in a `File`, hands to `uploadImageAsset`. Used by `Export` to durably re-host Higgsfield CloudFront URLs in our own Supabase bucket. Filename derives from URL tail with a `.png` fallback.
- **`createImageAssetFromUploaded()`** _(Slice 4)_ — `useAssetStore` action. Shortcut for `Export`: skips the file → upload step the caller already did, just commits an `ImageAsset` from the descriptor.
- **Moodboard** — an asset of kind "collection of images" used for visual reference. Distinct from `ImageIterator` in intent — moodboards convey style/vibe, iterators feed batches.
- **Recipe** — a saved subgraph (nodes + connections + their config) that can be re-instantiated on a canvas later. The unit of reuse.
- **Workflow** — the live state of nodes on the current canvas. Becomes a recipe when saved.
- **Project** — a workspace containing one or more workflows + a scoped set of project-specific assets (in addition to global library assets).

## Engine

- **Node** — a single unit on the canvas. Defined by a schema: `{ id, inputs, outputs, config, execute }`.
- **Reactive node** — runs automatically when upstream changes (e.g. `Text`, `Number`, `ArraySplit`). Cost: zero.
- **Executable node** — only runs when explicitly requested (e.g. `LLMText`, `HiggsfieldImageGen`). Cost: non-zero.
- **StandardizedOutput** — every node's output conforms to `{ type, format, data, metadata }`. Lets generic downstream consumption.
- **Run** — one execution of a target node (and the upstream nodes needed to satisfy it).
- **Layer** — a set of nodes in the topological sort that can execute in parallel.
- **Cache key** — `hash(nodeId + serializedConfig + sortedUpstreamOutputHashes + seedStrategy + lockedSeedValue)`. Determines cache hit/miss.
- **Seed strategy** — per-node config. `locked` (deterministic), `random` (every run different), `inherited` (follows recipe-level seed).
- **Pin** — flag on a specific output that makes it immune to cache invalidation. Unpin to reset.

## Engine

- **NodeSchema** — declarative shape of a node type: kind, category, inputs, outputs, defaultConfig, optional `execute`, `Body` component, optional `reactive` flag, optional `settings: { Content; hasOverrides? }` slot (ADR-0027 — when present, BaseNode renders a `⋯` trigger in the header that opens a popover with `Content`), optional `size: NodeSizeSchema` slot (ADR-0028 — when present, BaseNode applies the dim constraints and, if `resizable !== "none"`, renders a drag handle in the matching position).
- **NodeSizeSchema (ADR-0028)** — every field optional: `defaultWidth` / `defaultHeight` (initial size before any user resize; unset = content-driven), `minWidth` / `maxWidth` / `minHeight` / `maxHeight` (floor / ceiling applied to both content-driven natural size *and* user-resized size via NodeResizeControl drag bounds), `resizable: "none" | "horizontal" | "vertical" | "both"` (default `"none"`; when set, BaseNode renders the standardised drag handle in the matching position). LLM Text declares `both` (output area scrolls within `maxHeight: 520`); Text declares `both`; Image declares `horizontal` (the `aspect-square` preview makes vertical resize a no-op).
- **NodeInstance.size (ADR-0028)** — per-instance user-resized dimensions: `{ width?: number; height?: number }`. Either axis can be set independently (Image declares horizontal-only resize, so only `width` ever lands). When present, overrides `schema.size.defaultWidth` / `defaultHeight` for *this* instance only. Wired through `canvas-flow.tsx`'s `onNodesChange` → `useWorkflowStore.resizeNode()` on every drag end.
- **defineNode** — identity helper that produces a typed `NodeSchema<TConfig>`. Lives in `src/lib/engine/define-node.ts`.
- **NodeRegistry** — central catalog. `register(schema)` / `get(kind)` / `list()` / `listByCategory()`. The singleton `nodeRegistry` is populated by `all-nodes.ts` on import.
- **NodeInstance** — runtime occurrence of a schema on the canvas: `{ id, kind, position, config }`. Lives in the workflow-store.
- **WorkflowEdge** — `{ id, source, sourceHandle, target, targetHandle }`. Stored alongside NodeInstances in the workflow-store.
- **StandardizedOutput** — the only shape that flows through edges. Discriminated union over `text | image | video | number`. A node may emit a single value or an array (iterators).
- **DataType** — `"text" | "image" | "video" | "number" | "any"`. Handles carry a DataType for colored dots + connection compatibility.
- **NodeIO** — single handle descriptor: `{ id, label, dataType, multiple? }`.
- **NodeCategory** — `"input" | "iterator" | "ai-vision" | "ai-text" | "ai-image" | "ai-video" | "transform" | "compose" | "output"`. Drives AddNode popover grouping.
- **Reactive node** — output is a pure function of `config`; no upstream input needed (Text, Image, Number). The run engine treats reactive nodes as always-fresh sources.
- **Executable node** — has `execute()` + non-empty inputs; requires the run engine to be invoked (M0a Slice 3).
- **ExecContext** — `{ nodeId, config, inputs, signal }` passed to `execute`. The engine fills `inputs` with resolved StandardizedOutputs.
- **NodeBodyProps** — `{ nodeId, config, updateConfig, selected }`. The schema's Body component receives this to render the node interior.
- **BaseNode** — the shared card chrome (header + body slot + footer + colored handles) wrapping every schema's Body.
- **CanvasFlow** — React Flow mount that bridges workflow-store ↔ React Flow's internal model. One generic React Flow node type (`"cookbook"`) dispatches by schema kind.

## Run engine (M0a Slice 3.1)

- **runWorkflow** — `src/lib/engine/run-workflow.ts`. The whole evaluator. Pure function: graph + registry + cache + signal + `onProgress` callback in, per-node `ExecutionRecord` stream out. Serial topological walk. See ADR-0019 for the rationale.
- **ExecutionStatus** — `"idle" | "pending" | "running" | "done" | "cached" | "error" | "cancelled"`. The seven states a node can be in during/after a run. `idle` is the implicit pre-run state (the store doesn't even hold a record for it). `cached` is distinct from `done` so the UI can communicate "this was free" and the cost preview can exclude it.
- **ExecutionRecord** — `{ status, output?, error?, elapsedMs?, hash?, usage? }`. One per node per run. Lives in the execution-store's `records: Map<nodeId, ExecutionRecord>`. `usage` is populated for nodes that opt into the rich `{ output, usage? }` return shape (LLM Text in Slice 3.3; HiggsfieldImageGen / FalNanoBananaEdit in later slices).
- **NodeUsage** — `{ costUsd?, inputTokens?, outputTokens?, model? }`. Optional usage block a node attaches to its execution result; the engine extracts it into `ExecutionRecord.usage`. Every field is optional so partial reporting works (e.g. a future audio node that only knows duration). `model` is what *actually* ran — Fal occasionally re-routes — so the queue shows what got billed, not what was requested.
- **NodeExecuteResult** — discriminated union over the three legal return shapes from `execute()`: `StandardizedOutput`, `StandardizedOutput[]`, or `{ output, usage? }` (the `NodeOutputWithUsage` rich form). The runner collapses all three into `{ output, usage? }` at `normalizeExecuteResult`. Throws on anything else so node-author bugs surface immediately.
- **NodeOutputWithUsage** — `{ output: StandardizedOutput | StandardizedOutput[], usage?: NodeUsage }`. The rich return form. Recognised structurally (has an `output` field; `StandardizedOutput` has a `type` field instead) — no constructor / brand needed.
- **normalizeExecuteResult** — internal in `run-workflow.ts`. Turns each legal return shape into a uniform `{ output, usage? }` before the runner touches the cache or emits a record. Order matters: array first, then `type`-discriminated StandardizedOutput, then `output`-bearing rich form, then defensive throw.
- **Cache key (node hash)** — `fnv1a_64(stableStringify({ kind, config, deps }))` where `deps` is `[{ handle, sourceHash }, …]` sorted by `(handle, sourceHash)`. Same hash → same cached output. Editing a config field, swapping which input an edge feeds, or any upstream's hash changing all bust the key. See `computeNodeHash` + ADR-0019 for the literal recipe.
- **stableStringify** — recursive JSON stringify with sorted object keys at every depth (arrays kept in order). Pair only with `hashString`; mixing them with `JSON.stringify` would silently desync the cache.
- **Topological sort** — Kahn's algorithm in `topologicalSort`. Stable: ties broken by original node order. Dangling edges (referencing missing nodes) are ignored defensively. Cycles → `hasCycle: true` and the engine flips every node to `error` with `"Cycle detected in workflow"`.
- **Session cache** — `Map<hash, ExecutionCacheEntry>` where `ExecutionCacheEntry = { output, usage? }`. Lives at module scope in `execution-store.ts`. In-memory only; reload = empty. Persistent caching is a Slice 5 concern (SQLite alongside the workflow). The usage block is persisted in the cache so a cache hit replays the same `costUsd`/`tokens` into the new record (otherwise re-runs would credit cached calls as free, lying about what the workflow would have cost).
- **execution-store** — `src/lib/stores/execution-store.ts`. Zustand store holding `{ runId, isRunning, records }`. Actions: `startRun()` (preempts any active run; bumps `runId`; uses `runId` guard to drop stale callbacks), `cancelRun()` (aborts the active `AbortController`), `clearRun()` (wipes records, keeps cache), `clearCache()` (drops cache, keeps records). `getRecord(id)` is the recommended accessor — Map identity changes on every set, so subscribing to a `.get(id)` selector is the right granularity. Use `_resetExecutionForTests()` to reset between tests.
- **runId guard** — every `startRun` bumps `runId`; progress callbacks check `useExecutionStore.getState().runId === runId` before mutating records. Late callbacks from a cancelled run get dropped silently.
- **NodeStatusChip** — `src/components/nodes/status-chip.tsx`. The tiny icon in the BaseNode header that reflects the node's `ExecutionStatus`. Subscribes only to its own node's record so unrelated chips don't re-render on every progress emission. Idle = renders nothing. See ADR-0020.
- **RunButton** — `src/components/layout/run-button.tsx`. Top-right chrome pill between Gallery and AddNode. Two states (idle = accent Play, running = spinner + Square + "Cancel"). Disabled when the graph is empty.
- **LLMText (Slice 3.2 — real Fal OpenRouter wiring)** — `src/components/nodes/node-llm-text.tsx`. `kind: "llm-text"`, `category: "ai-text"`. Three input handles (`user` text + multi, `system` text, `image` image + multi), one `out` text output, config `{ model }`. Body is the ADR-0023 chip + output area, with a third state in between — when `status === "error"` the body renders a destructive-tinted `role="alert"` pill with the error message inline (so users don't have to hover the 12 px status chip to learn what broke). `execute()` joins multi-edge `user` inputs with blank lines, takes the single `system` text, extracts `image.url`s from any wired image refs, and calls `callOpenRouter` — the server picks `openrouter/router` or `openrouter/router/vision` based on whether images were sent. Returns `{ type: "text", value: result.text }`. Throws when `user` is empty (no inline fallback) so the requirement to wire upstream Text nodes is loud and obvious. Honors the engine's `AbortSignal` end-to-end via fetch.
- **Gemini 2.5 Pro reasoning requirement** — `google/gemini-2.5-pro` is back in the curated MODEL_OPTIONS list as of Slice 3.4, tagged `reasoningRequired: true`. Fal's `openrouter/router` rejects Pro without `reasoning: true` ("Reasoning is mandatory for this endpoint and cannot be disabled"). The LLM Text settings popover surfaces an accent-coloured hint when the user picks Pro without ticking reasoning, so the misconfiguration is caught at config time instead of failing the run with an upstream 400. `gemini-2.5-flash` stays in the list as the cheaper non-reasoning default. The same `reasoningRequired` flag will tag o-series and other reasoning-mandatory models as they land.

## LLM call path (M0a Slice 3.2)

- **`/api/fal/openrouter` route** — `src/app/api/fal/openrouter/route.ts`. POST handler. Reads + Zod-validates the JSON body (`llmRequestSchema`), calls the server wrapper with the request's `AbortSignal`, returns 200 + `LlmSuccessResponse` on success or `{ error, code }` on failure. Standard error codes: `invalid_request` (400, body validation failed — message is prefixed with the failing field path), `aborted` (499, client disconnected or wrapper threw AbortError), `missing_key` (500, FAL_KEY not in server env), `upstream_error` (502, Fal returned a structured error), `unknown` (500, anything else — generic message; full error logged to the Next terminal). `dynamic = "force-dynamic"` + `runtime = "nodejs"` so the route always executes and the Fal SDK gets a full Node fetch/stream stack.
- **`callFalOpenRouter()` (server wrapper)** — `src/lib/llm/fal-openrouter.ts`. `import "server-only"`-guarded. Lazy-configures the Fal client (`fal.config({ credentials: process.env.FAL_KEY })`, cached after first call), picks `openrouter/router/vision` if `images.length > 0` else `openrouter/router`, builds the per-endpoint input inline (omitting `undefined` keys — Fal is strict on null fields for some models), races `fal.subscribe(...)` against the AbortSignal so cancelled runs reject immediately, and annotates errors with `code` (`missing_key`, `upstream_error`) before throwing. Returns `{ text, model, costUsd?, inputTokens?, outputTokens? }`.
- **`callOpenRouter()` (client wrapper)** — `src/lib/llm/call-openrouter.ts`. Browser-side fetch wrapper. Forwards the `AbortSignal` to fetch so engine cancellation is honored end-to-end; re-throws fetch's `AbortError` unchanged and translates the server's `499` response into an `AbortError` too so the engine routes cancelled runs into `cancelled` (not `error`); maps non-OK JSON responses into `LlmCallError(code)` with the server's message; maps non-JSON failure bodies into a generic message; maps fetch-level network failures into `LlmCallError("network")`.
- **`LlmCallError`** — error subclass thrown by the client wrapper. `.code` is one of the server's `LlmErrorResponse["code"]` values plus `"network"` for fetch failures. Use `instanceof LlmCallError` to distinguish from `AbortError` (cancellation).
- **`llmRequestSchema`** — Zod schema in `src/lib/llm/types.ts`. Single source of truth shared by the route validator and the client typing. Shape: `{ model: string, user: string, system?: string, images?: string[] (urls), temperature?: number 0-2, maxTokens?: positive int, reasoning?: boolean }`. The optional generation knobs land in 3.4 alongside the LLM Text settings popover; the server wrapper spreads each one into the Fal `subscribe` input only when defined so the provider's default kicks in for unset fields.
- **`tests/shims/server-only.ts`** — empty module that `vitest.config.ts` aliases `"server-only"` to, so server-only modules can be imported in unit tests without tripping the build-time guard. Add new server modules here for free — no per-file change needed.

## AssetGroup model (Slice 5.6 / ADR-0032)

- **`AssetGroupAsset`** — third kind on the `Asset` union, joining `image` and `soul-id`. Shape: `{ kind: "asset-group"; assetIds: string[]; isUntitled: boolean }`. Pure metadata, no bytes — the underlying `image` ids survive group deletion. Image-only and flat (no nesting / no cross-kind) for M0a. The library's "Groups" section renders one card per `AssetGroup`.
- **`isUntitled`** — flag on `AssetGroupAsset` distinguishing groups created automatically (multi-image drag from library, v8 → v9 migration) from groups the user explicitly named. `true` is the auto-cleanup trigger: deleting an iterator that owns an Untitled group with no other linkers drops the group from the library. `renameGroup` flips this to `false` permanently — the user just told us "this is a real group worth keeping". The visual badge ("Untitled" pill on the card) reminds the user they can rename to promote.
- **Group subview** — clicking (or programmatically opening via `onOpen`) a group card flips `LibraryContent` into a sub-view that shows the group's image members in the same 2-col grid, with a back arrow + the group name (also editable inline) + a count badge. Subview state is purely local React state — no store change. If the active group gets deleted while the subview is open (cleanup ran), the next render bounces to top-level via the `activeGroup === null` fall-through.
- **Iterator linked to a group (`config.groupId`)** — every Image Iterator on the canvas is **always linked** to an `AssetGroup` via its config (Slice 5.5's free-floating `assetIds[]` is gone). Empty string is a transient placeholder for the moment between `addNode()` and the dispatcher's `groupId` write. The body shows a "no group linked" empty state until then. `execute()` resolves through `useAssetStore.getState().getAsset(groupId).assetIds` at runtime, then through the standard image refs. Deleted / missing groups → empty array, fan-out branch ends gracefully.
- **Detach from group** — settings popover button on the iterator. Creates a NEW `AssetGroup` named `"<source> (copy)"` with the SAME image ids (no byte duplication), then `updateConfig({ groupId: newGroupId, cursor: 0 })`. Mirrors Figma's "Detach instance" + Photoshop's "Smart Object → Layers". The source group is preserved verbatim — useful when you want to fork a recipe without affecting the original.
- **`createGroup({ name?, assetIds, isUntitled?, scope? })`** — asset-store action that creates an `AssetGroupAsset` with de-duped, first-seen-order `assetIds`. Auto-names `"Untitled <N>"` (counts existing Untitled to keep the sequence growing). `isUntitled` defaults to false; the four legitimate `true`-callers are: Slice 5.6c "Import as group" with empty name, Slice 5.6d multi-image drag dispatcher, Slice 5.6a workflow-store v8→v9 migration, and the canvas-flow `create-group-and-spawn-iterator` action handler.
- **`addToGroup(groupId, assetIds)` / `removeFromGroup(groupId, assetIds)`** — append (de-duped against existing) / filter actions on a group's `assetIds`. `updatedAt` bumps on success; downstream subscribers (multiple iterator views on the same group) re-render naturally. Empty inputs are no-ops.
- **`renameGroup(groupId, name)`** — trims `name`; ignores empty / whitespace-only. On first non-empty rename flips `isUntitled` to `false` permanently — the cleanup rule no longer applies.
- **`removeGroup(groupId)`** — drops the group record from the asset store. Underlying `image` assets survive (they're the durable thing). Defensive scrub from `selectedAssetIds` so a group selected in the library when deleted doesn't haunt the next shift-click.
- **`cleanupUntitledGroupIfOrphan(groupId, linkedNodeIds)`** — asset-store action; the only group-deletion path triggered by canvas events. Drops the group iff `isUntitled === true` AND `linkedNodeIds.length === 0`. Caller computes the linked-iterator set from the workflow store; keeps the asset store decoupled from the workflow store. No-ops on missing / non-Untitled / linked groups (all idempotent).
- **`cleanupGroupIfOrphan(groupId)` (canvas glue)** — `src/lib/library/cleanup-orphan-group.ts`. Walks `useWorkflowStore.getState().nodes` for image-iterators with `config.groupId === groupId`, calls `cleanupUntitledGroupIfOrphan` with the resulting set. Wired into both node-removal paths in `canvas-flow.tsx` (Backspace handler + RF `onNodesChange` `c.type === "remove"`).
- **Workflow-store v8 → v9 migration** — for every iterator with `assetIds[]` in its config: materialise an `Untitled` group via `useAssetStore.getState().createGroup(...)`, rewrite the iterator's config to `{ groupId, cursor, selectionMode, range? }`. Selection mode + cursor + range carry over verbatim. Idempotent on v9 payloads. **Requires asset-store rehydrated FIRST** — `shell.tsx`'s `useEffect` enforces this (asset-store `persist.rehydrate()` is called before `workflow-store.persist.rehydrate()`).
- **Asset-store v4 → v5 migration** — additive (the `asset-group` kind didn't exist pre-v5). Defensive sweep drops malformed `asset-group` rows (missing `assetIds[]` or `isUntitled` boolean) so hand-edited localStorage can't crash the iterator's runtime resolver.
- **`importImageFilesAsGroup(files, name)`** — `src/lib/library/import-files.ts`. Wraps `importImageFiles` + `createGroup`. Returns `{ ...originalResult, groupId: string | null }`. All-failed → `groupId: null`, no group created. Empty / whitespace name falls back to "Untitled". Used by the import-as-group dialog (Slice 5.6c).
- **`ImportAsGroupDialog`** — `src/components/library/import-as-group-dialog.tsx`. Controlled by `files: File[] | null` prop. Two actions ("Import as N separate" / "Import as group named [...]") + Cancel. Body sub-component mounts only while open so input state is fresh on every selection (no `useEffect`-based sync).
- **`dispatchAssetDrop` action variants (Slice 5.6d, supersedes 5.5c)** — `src/lib/library/dispatch-asset-drop.ts`. Pure / framework-agnostic decision tree. Variants: `spawn-node`, `create-group-and-spawn-iterator` (replaces 5.5's "spawn iterator with assetIds[]"), `append-to-group` (replaces "append-to-iterator"; operates on the iterator's linked group via `addToGroup`), `noop`. Group payload routing: drag of a group on canvas → spawn iterator linked to it; drag of a group on existing iterator → `append-to-group` with `@group:<id>` sentinel that the canvas-flow caller expands through the asset store.
- **`@group:<id>` sentinel** — a string convention used by the dispatcher to communicate "this is a group id; expand it through the asset store before calling addToGroup". Local to one emitter (`dispatchAssetDrop`'s `asset-group → existing iterator` branch) and one consumer (`canvas-flow.tsx`'s `append-to-group` action handler). Avoids leaking the asset store into the dispatcher (which would couple two layers). If the protocol spreads, it gets a Zod schema.

## Projects & documents (M1 projects arc, ADR-0049 + ADR-0050)

- **ProjectDocument** — `src/lib/project/document.ts`. The single canonical, versioned serialization of a project (graph + assets + layout + `executionState` + name). - **`.cookbook` file / bundle** — `src/lib/project/file.ts`. `.cookbook` = JSON document (media as URLs); `.cookbook.zip` = self-contained bundle (`project.json` + a `media/` folder of bytes via `fflate`, URLs rewritten to relative paths). `importProjectFile` reads either; `importProjectToCloud` opens one into a NEW cloud project (re-hosting bundle media to the user's bucket).

## Cookbook Library (Phase A — 2026-06-01)

See [`docs/COOKBOOK-LIBRARY.md`](./COOKBOOK-LIBRARY.md) for the full design + roadmap. This glossary entry is the quick reference.

- **Cookbook** — full-screen overlay opened via the top-nav book button (or ⌘B). Tabs: Recipes (browse / inspect / manage saved subgraphs) and Prompts (every prompt the app uses, with plain-English descriptions). Distinct from the asset *Library* (images / videos / audio). - **Recipe (Cookbook sense)** — a saved subgraph that acts as a composite node. Two flavors: *system* (`owner_id IS NULL`, built into the app, visible to everyone, can't be edited but can be duplicated) and *user* (`owner_id = me`, visible only to me, editable in Phase B). Same `RecipeRecord` interface either way, just different ownership.
- **Prompt (Cookbook sense)** — a block of text instructions sent to an LLM. Four kinds tracked by the Library: *Assistant base* (`REASONER_INSTRUCTIONS`, lives in code at `src/lib/assistant/instructions.ts`), *Specialist role* (Phase D — code-defined overlays), *Recipe internal* (Text-node bodies + LLM call meta inside a recipe's subgraph), *Node default* (hardcoded starter text in node specs).

Composite-node instances on a canvas may carry a `recipeVersion` so the canvas can show "Update available → vN+1" badges. Distinct from `RecipeSubgraph.version` (the schema-shape version of the subgraph payload).
- **Override (Phase C)** — a user's personal customization of a code-defined prompt (assistant base, specialist role). Stored in `app_prompt_overrides` keyed by `(user_id, prompt_key)`. Doesn't change the prompt for anyone else. Always reversible by clicking "Restore default" — that just deletes the row.
- **`PromptEntry`** — `src/lib/prompts/types.ts`. `{ key, title, description, section, content, recipeId?, recipeName?, internalNodeId?, internalNodeKind?, purpose? }`. The shared shape consumed by the Prompts tab regardless of whether the prompt came from the registry or a recipe extraction.
- **`getCodePrompts()`** — `src/lib/prompts/registry.ts`. Returns the Library's static list of code-defined prompts. Phase A: just the assistant base prompt. Phase D: append role overlays.
- **`extractRecipePrompts(recipe)`** — `src/lib/prompts/extract-from-recipe.ts`. Pure function. Walks a recipe's subgraph for `text` nodes (their `text` config is the prompt body) and `llm-text` nodes (meta entry — model + temperature). Skips Text bodies < 16 chars (placeholders). Infers `purpose` ("system prompt" / "user prompt" / "system prompt fragment" via one-hop text-concat chain follow / etc.) from how the node is wired downstream.
- **Cookbook overlay** — `src/components/cookbook/cookbook-overlay.tsx`. 94vh × 96vw centered modal. Esc / backdrop-click / explicit close button all close. Tabs persist via `useLayoutStore.cookbookTab`. Recipes tab + Prompts tab both render side-by-side list-and-detail panels.

Returns `{ spawned, imported, errors, skipped }`. Every dependency injectable for tests (no Supabase / no Zustand needed). Default fan-out offset is +24/+24 px per spawned node, matching the Library / Gallery drop conventions.
- **`extractImagesFromClipboard(data)`** — `src/lib/canvas/handle-canvas-paste.ts`. Pure helper that walks `DataTransfer.files` first (modern browsers), then falls back to `DataTransfer.items` for Safari and a few WebKit-derived browsers. Returns image `File[]` only — non-image clipboard content is intentionally ignored (paste is image-only by design; drop is the multi-media path).
- **`isEditablePasteTarget(target)`** — `src/lib/canvas/handle-canvas-paste.ts`. Same input/textarea/contentEditable guard the keyboard clipboard handler uses, locally duplicated to keep the two surfaces independent.
- **Canvas `onDrop` external-files branch** — `src/components/canvas/canvas-flow.tsx`. Fires when none of the in-app MIMEs (`ASSET_DRAG_MIME` / `GENERATION_DRAG_MIME` / `RECIPE_DRAG_MIME`) are present but `dataTransfer.files` is non-empty. Drop position uses `screenToFlowPosition` so the first node lands exactly under the cursor. Toasts use the result envelope from `handleExternalFilesDrop`.
- **Canvas `paste` listener** — `src/components/canvas/canvas-flow.tsx`. Document-level listener; calls `extractImagesFromClipboard` + `handleExternalFilesDrop` at `getSpawnPosition()` (viewport center). Skips when focus is on an editable target. Falls through (no `preventDefault`) when the clipboard has no image content so the existing node-clipboard ⌘V handler still receives the event.

## Image generation nodes: grid ↔ single preview (2026-06-02)

- **`MultiImageView`** — `src/components/nodes/multi-image-view.tsx`. Shared multi-image preview surface for image generation nodes (Fal Image, Higgsfield Image Gen). Two view modes — `"grid"` (2-col clickable thumbnails, click flips to single mode focused on that index) and `"single"` (one large image at `previewIndex` + bottom overlay strip with back-to-grid button + `IteratorCursor`). 1-image batches short-circuit to a plain `MediaPreviewImage` (no overlay); 0-image batches render nothing (caller owns empty state). `previewIndex` is clamped on render so re-runs returning fewer images can't crash. Optional `gridTileAspectRatio` lets nodes keep a curated tile aspect (e.g. Higgsfield's `"1 / 1"`) different from the single-mode aspect.

- **`Video Pad` node (`video-pad`)** — `src/components/nodes/node-video-pad.tsx`. Single video-in / single video-out transform that extends a short clip to a minimum duration by holding the first or last frame. Default `minDurationSec: 4` (the floor most video-understanding LLMs enforce — Marlin, Scribe-v2, several Fal endpoints), default `padMode: "end"` (hold the last frame after the clip — reads as a held beat, not a synthetic preroll). `padMode: "start"` holds the first frame before the clip; `padMode: "both"` splits the deficit evenly with the odd millisecond biased to the end. Audio is dropped (matches `video-slicer`). Two passthrough fast paths: `minDurationSec <= 0` and "source already meets the minimum" both emit the source URL unchanged with no upload. Otherwise re-encodes through `padVideoToMinDuration` and uploads through `uploadMediaAsset(file, "videos")`.

- **`padVideoToMinDuration`** — `src/lib/media/pad-video.ts`. Mediabunny single-pass re-encoder backing the Video Pad node. Walks one MP4 output backed by a `CanvasSource` (codec `"avc"`, `QUALITY_HIGH`, source dimensions) across three stages on the same `OffscreenCanvas`: (a) draw the first decoded `VideoSample` and emit `holdFps` samples for `padStartMs`; (b) iterate `sink.samples()` and re-encode every source sample with `timestamp += padStartSec`; (c) draw the last sample (probed at `duration - 0.05s`) and emit `padEndMs` of held samples. Returns `{ blob, sourceDurationMs, paddedDurationMs, padStartMs, padEndMs }`. Fast path: when `sourceDurationMs >= minDurationMs`, returns `{ blob: null, ... }` so the caller can pass the source URL through with zero re-encode + upload.

- **`splitPadDuration`** — `src/lib/media/pad-video.ts`. Pure helper that distributes the missing duration between start and end pads according to `PadMode`. `"start"` puts the whole deficit at the start; `"end"` at the end; `"both"` splits in half, biasing the odd millisecond to the end so `source + padStart + padEnd` is exact. Returns zeros when source already meets the minimum. Exported separately so node UIs and tests can preview the split without invoking mediabunny.

- **`viewMode` / `previewIndex` (FalImage / Higgsfield config)** — Two optional fields on `FalImageNodeConfig` and `HiggsfieldImageGenNodeConfig`. `viewMode: "grid" | "single"` toggles the preview shape; `previewIndex: number` is the focused image in single mode (0-based). Persisted on `config` like `Text.previewMode` / `List.cursor` / `ImageIterator.cursor` — no separate `NodeInstance.uiState` container. View-only: doesn't touch `record.output`, so downstream nodes still see the full image array regardless of which image is currently focused in the UI.
- **Two-cursor composition** — On a generator that has both N>1 generations in history AND N>1 images in the current batch, two `IteratorCursor`s render in different corners. Top-right (`absolute right-1 top-1`) navigates past runs (existing `useNodeHistoryCursor`); bottom (`absolute inset-x-1 bottom-1`, only in single mode) navigates images inside the current run (new `MultiImageView`). Same visual chip, different scopes — no shared state.

## Process

- **Test-as-you-go** — every shipped feature lands with at least one automated test + a manual smoke test from the user before the next feature starts.
- **Pause-verify-continue** — agent stops at natural milestones (visible deliverable), shows the user (screenshot + how-to-test), waits for confirmation, then continues.
